Cardre scorecard flow

Basic scorecard flow, stage by stage

This report follows the canonical fixed scorecard pathway from the Phase 2 technical spec and traces the build stream from dataset import through score scaling, then the validation stream that applies the fitted artefacts to train, test, and oot samples.

The report focuses on the artefacts Cardre actually writes: Parquet datasets, JSON definitions and reports, model JSON, scorecard JSON, and the run manifest. Alternative launch nodes such as cardre.auto_binning_fit are omitted because they are not part of the canonical pathway described here.

What to watch for

  • Build streamFits binning, WOE, model, and scorecard parameters on train only.
  • Validate streamApplies the fitted definitions to train, test, and oot.
  • Artefact typesDatasets are Parquet; definitions, reports, models, scorecards, and manifests are JSON.
  • BoundaryScore scaling finalises the build stream; validation starts after that point.

Artefact legend

These are the payload families that recur throughout the flow.

Dataset

Parquet rows and columns, role-tagged as input, train, test, oot, or scored outputs.

Definition

JSON contracts for modelling metadata, sample definition, bin definitions, selection, and scorecard bundle metadata.

Report

JSON or Parquet analytics: profiles, WOE/IV, clustering, validation, cutoff analysis, and build summaries.

Model / scorecard

JSON payloads that persist fitted coefficients and the score translation from log-odds to points.

Build stream

Everything from import through score scaling. All fitting happens on train.

0Import datasetcardre.import_dataset
InputsAbsolute source_path, format hints, delimiter, header flag, encoding, null values, schema overrides, and optional row limit.
TransformationReads CSV, TSV, or Parquet into a Polars frame, infers or applies schema, and normalises the dataset into Cardre's internal import shape.
OutputsOne Parquet dataset with role input.
Artefact data
  • Payload columns and rows from the source file.
  • Metadata: source_file, format, columns, row_count.
  • Optional metadata: max_rows_applied; optional warning when truncation occurs.
Sourcecardre/nodes/prep.py
1Define modelling metadatacardre.define_modelling_metadata
InputsImported or filtered dataset and the modelling contract fields.
TransformationChecks that the target exists, that good and bad classes are non-empty and disjoint, and that observed values are fully declared.
OutputsOne definition JSON artefact.
Artefact data
  • target_column, good_values, bad_values, indeterminate_values.
  • Business context: population, product, segment, observation_window, performance_window.
  • schema_version = cardre.modelling_metadata.v1.
Sourcecardre/nodes/prep.py
2Apply exclusionscardre.apply_exclusions
InputsDataset plus optional modelling metadata and a list of exclusion rules.
TransformationEvaluates each rule, filters matching rows out of the dataset, and records how many rows each rule removed.
OutputsA filtered dataset with the same role as the input, plus an exclusion report.
Artefact data
  • Dataset metadata: source_artifact_id, rows_before, rows_after.
  • Report fields: rows_before, rows_after, rows_excluded, and per-rule entries with column, operator, value, reason, rows_removed.
Sourcecardre/nodes/prep.py
3Profile datasetcardre.profile_dataset
InputsAny dataset role; in the canonical build path this is the filtered dataset from the exclusion step.
TransformationCalculates a column inventory, dtype map, null counts, and simple numeric summaries.
OutputsOne report JSON artefact.
Artefact data
  • row_count, column_count, columns, dtypes, null_counts.
  • numeric_stats per numeric column with min, max, mean, and std.
  • profile_steps is present as an empty list in the current implementation.
Sourcecardre/nodes/prep.py
4Validate binary targetcardre.validate_binary_target
InputsFiltered dataset and the target column name.
TransformationChecks that the target has exactly two unique non-null values and records the observed class counts.
OutputsOne report JSON artefact.
Artefact data
  • target_column, unique_values, count, is_binary, value_counts, null_count.
  • The step fails if the value count is not 2.
Sourcecardre/nodes/prep.py
5Define development samplecardre.development_sample_definition
InputsDataset or train subset and sample-domain controls.
TransformationWrites the development-sample contract without sampling the rows yet; this is a declaration step, not a split step.
OutputsOne definition JSON artefact.
Artefact data
  • sample_method, sample_domain, total_rows, financed_rows, non_financed_rows.
  • Reject/approval controls: rejection_source, rejection_column, rejection_values, approval_column, approval_values.
  • Optional weighting and narrative: weight_column, sample_description, population_bad_rate, prior_probability_adjustment.
Sourcecardre/nodes/prep.py
6Split train / test / ootcardre.split_train_test_oot
InputsDataset and sample definition; either a stratified split or a preassigned role column.
TransformationCreates three role-tagged subsets and preserves the target mix where possible.
OutputsThree Parquet datasets with roles train, test, and oot, plus a split report.
Artefact data
  • Each dataset stores source_artifact_id, strategy, and row_count in metadata.
  • Split report: strategy, random_seed, fractions, row_counts, target_rates, warnings, source_artifact_id.
Sourcecardre/nodes/prep.py
7Explicit missing / outlier treatmentcardre.explicit_missing_outlier_treatment
InputsRole datasets plus treatment maps for imputations, caps, and floors.
TransformationApplies the requested replacements and numeric limits to each role dataset, then records what changed.
OutputsThree treated role datasets and a treatment report.
Artefact data
  • Dataset metadata: source id and a compact list of treatment keys applied to the role.
  • Report payload: nested imputations, caps, and floors maps keyed by column name with counts, values, and reasons.
Sourcecardre/nodes/prep.py
8Fine classingcardre.fine_classing
Inputstrain plus modelling metadata.
TransformationBuilds initial numeric and categorical bins, counts goods and bads in each bin, and emits warnings for sparse bins or high cardinality.
OutputsOne definition JSON artefact holding the bin definition.
Artefact data
  • Top-level payload: schema_version, variables, warnings.
  • Each variable: variable, kind, bins.
  • Each bin: bin_id, label, lower, upper, inclusion flags, categories, is_missing_bin, row_count, good_count, bad_count.
Sourcecardre/nodes/build/bins.py and cardre/engine/binning/definition.py
9Calculate WOE and IVcardre.calculate_woe_iv
Inputstrain, the current bin definition, and modelling metadata.
TransformationComputes bin-level WOE and variable-level IV, applies optional additive smoothing, and can reject non-monotonic final WOE when configured.
OutputsFour report artefacts: WOE table, IV ranking, summary, and controlled evidence JSON.
Artefact data
  • WOE table: variable, bin_id, label, row_count, good_count, bad_count, good_distribution, bad_distribution, woe, iv_component.
  • IV table: variable, iv, bin_count, zero_cell_count, warning_count.
  • Summary: purpose, zero-cell policy, smoothing config, formula, and warnings.
  • Evidence: per-variable status plus per-bin diagnostics and smoothing details.
Sourcecardre/nodes/build/features.py and cardre/_evidence/models.py
10Variable clusteringcardre.variable_clustering
Inputstrain and IV evidence, optionally WOE-transformed representations.
TransformationBuilds pairwise correlations, groups redundant variables into clusters, and picks a representative suggestion per cluster.
OutputsOne clustering report JSON artefact.
Artefact data
  • Config: method, input_representation, similarity_metric, threshold, missing_handling, candidate_limit, minimum_pair_count, representative_rule.
  • Results: clusters, singleton_variables, warnings.
  • Cluster members carry variable, iv, and missing_rate alongside the representative suggestion.
Sourcecardre/nodes/build/clustering.py and cardre/_evidence/models.py
11Variable selectioncardre.variable_selection
InputsIV report and optional clustering evidence.
TransformationRanks variables by IV, applies manual include/exclude rules, and optionally limits each cluster to one representative.
OutputsOne definition JSON artefact.
Artefact data
  • Top level: schema_version, min_iv, max_variables, cluster_representative_rule.
  • selected and rejected lists with variable names and reasons.
  • Optional cluster_decisions entries describing which representative won for each cluster.
Sourcecardre/nodes/build/selection.py and cardre/_evidence/models.py
12Manual binningcardre.manual_binning
InputsBin definition plus the variable-selection definition.
TransformationApplies validated bin overrides such as merges, grouped categories, reordering, or variable rejection.
OutputsA refined definition JSON artefact using the same bin-definition schema as fine classing.
Artefact data
  • The output keeps the same cardre.bin_definition.v1 shape.
  • Overrides are validated against the selected-variable set, and the result carries the updated bin structure plus any lifecycle history fields present in the definition model.
Sourcecardre/nodes/build/bins.py and cardre/engine/binning/definition.py
13Final WOE and IVcardre.calculate_woe_iv
InputsManual bins, train, and modelling metadata.
TransformationRecomputes WOE and IV after bin refinement. This is the production pass that the model should consume.
OutputsThe same four report artefacts as stage 9, now based on the final bins.
Artefact data
  • Same field set as the initial WOE/IV run.
  • In final mode, zero-cell smoothing and monotonicity enforcement matter more because the final report is the one linked to the scorecard.
Sourcecardre/nodes/build/features.py
14Technical manifest stubcardre.technical_manifest_export
InputsThe run steps and the evidence available so far in the build stream.
TransformationCollects run-step metadata, lineages, hashes, and the high-level modelling evidence available before validation starts.
OutputsOne manifest JSON artefact.
Artefact data
  • project, run, steps, artifacts.
  • Embedded modelling evidence: modelling_metadata, selected_variables, model, scorecard, validation_metrics, cutoff_analysis.
  • Also includes warnings and errors.
Sourcecardre/nodes/build/export.py and cardre/_evidence/models.py
15WOE transform traincardre.woe_transform_train
Inputstrain, final bin definition, WOE table, and optional selection definition.
TransformationAdds one _woe column per selected variable, checks that every bin has a WOE value, and rejects rows that do not map cleanly to a bin.
OutputsA WOE-transformed train dataset and a transform report.
Artefact data
  • Dataset metadata: source_artifact_id, woe_columns, target_column.
  • Report: target_column, transformed_variables, selected_only, row_count.
  • The transformed dataset now contains the original columns plus the fitted WOE columns.
Sourcecardre/nodes/build/features.py
16Logistic regressioncardre.logistic_regression
InputsWOE-transformed train dataset and modelling metadata.
TransformationFits a logistic regression on the WOE columns, preserves class orientation, and records convergence information.
OutputsOne model JSON artefact.
Artefact data
  • Identity and shape: model_family, target_column, features, source_variables.
  • Coefficients: intercept, coefficients, class_mapping, bad_class_label, target_event_value, probability_column_index.
  • Contract and training: feature_contract, feature_order_hash, training (row count, convergence, iterations, params), and warnings.
Sourcecardre/nodes/build/models.py and cardre/_evidence/models.py
17Score scalingcardre.score_scaling
InputsModel, final bin definition, WOE table, and score-scaling parameters.
TransformationConverts model log-odds into score points using the chosen base score, base odds, and points-to-double-odds factor. Each bin gets a point contribution derived from its coefficient and WOE.
OutputsOne scorecard JSON artefact.
Artefact data
  • Global score settings: base_score, base_odds, points_to_double_odds, factor, offset, higher_score_is_lower_risk, intercept, base_points, target_column.
  • attributes array with one entry per variable/bin: variable, bin_id, label, woe, coefficient, points.
  • schema_version = cardre.score_scaling.v1.
Sourcecardre/nodes/build/models.py and cardre/_evidence/models.py
18Build summary reportcardre.build_summary_report
InputsModel, scorecard, and the final WOE/IV artefacts.
TransformationProduces a compact narrative summary for the fitted model, scorecard, and supporting evidence.
OutputsOne report JSON artefact.
Artefact data
  • model_summary: target, features, intercept, coefficient count, convergence, row count.
  • scorecard_summary: base score, base odds, points to double odds, attribute count, score direction.
  • woe_iv_references: source artefact ids, row counts, and columns for IV and WOE tables.
  • warnings copied from the model artefact.
Sourcecardre/nodes/build/models.py

Governed packaging

Optional bundle creation that freezes the fitted scorecard into a single application artefact.

19Freeze scorecard bundlecardre.freeze_scorecard_bundle
InputsDefinition, report, model, and scorecard artefacts.
TransformationValidates that the model, WOE table, bin definition, and scorecard all line up, then packages them into one governed bundle.
OutputsOne scorecard JSON artefact whose payload is a scorecard application bundle.
Artefact data
  • Payload: schema_version, bundle_type, created_from, target, components, feature_contract, score_scaling, warnings.
  • Sidecar metadata: source artefact ids for model, scorecard, bin definition, WOE table, optional selection artefact, plus feature_count.
  • The bundle is what the validate stream can treat as the governed application package.
Sourcecardre/nodes/build/freeze.py

Validation stream

After the scorecard is finalised, Cardre applies the same fitted definitions to train, test, and oot.

20Apply WOE mappingcardre.apply_woe_mapping
InputsRole datasets, final bin definition, WOE table, and optional selection or frozen bundle.
TransformationBuilds WOE columns for each selected variable on each role dataset and enforces the unmatched-row policy.
OutputsWOE-applied train, test, and oot datasets plus an evidence report.
Artefact data
  • Dataset metadata: source artifact id of the input role.
  • Evidence payload: policy, roles, warnings, and references to the bin definition, WOE table, optional selection artefact, and optional frozen bundle.
  • Per-role evidence includes row_count, variables_applied, woe_columns_created, and unmatched-row counts.
Sourcecardre/nodes/validate/apply.py and cardre/_evidence/models.py
21Apply modelcardre.apply_model
InputsRole datasets, the fitted model, and optionally the scorecard or frozen bundle.
TransformationCalculates predicted bad probability for every row. If a scorecard is present, it also converts probability or log-odds to a score.
OutputsScored role datasets plus a score application evidence report.
Artefact data
  • Scored datasets keep the original columns and add predicted_bad_probability, raw_model_output, model_artifact_id, model_family, and optionally score and cardre_scaled_score.
  • Evidence payload: model_artifact_id, roles, warnings, and optional references to the scorecard or frozen bundle.
  • Per-role entries record source and output ids, required features, missing features, output columns, and probability/score summary stats.
Sourcecardre/modeling/adapters.py and cardre/nodes/validate/apply.py
22Validation metricscardre.validation_metrics
InputsScored role datasets and modelling metadata.
TransformationComputes discrimination, calibration, and stability metrics, then raises gates when configured thresholds are violated.
OutputsOne validation metrics report JSON artefact.
Artefact data
  • target with target column and good/bad labels.
  • roles with per-role metrics: row_count, bad_count, good_count, auc, gini, ks, ks_at_score, calibration, calibration_display, score_distribution, at_cutoffs, warnings.
  • stability with PSI values for train-vs-test and train-vs-oot, plus PSI warnings.
  • gates and top-level warnings.
Sourcecardre/nodes/validate/analyse.py and cardre/_evidence/models.py
23Cutoff analysiscardre.cutoff_analysis
InputsScored datasets and modelling metadata.
TransformationBuckets score ranges, then measures approval rate, bad rate, and capture rate per band or explicit cutoff list.
OutputsOne cutoff analysis report JSON artefact.
Artefact data
  • cutoff_tables keyed by role.
  • Each row contains score_cutoff, approval_rate, bad_rate, and capture_rate.
  • Optional warnings are emitted when the target metadata is missing or unhelpful.
Sourcecardre/nodes/validate/analyse.py
24Extended technical manifest exportcardre.technical_manifest_export
InputsThe full run history, including validation metrics and cutoff analysis.
TransformationRebuilds the run manifest with the complete evidence set, hashes, and lineage for the finished build and validation run.
OutputsOne final manifest JSON artefact.
Artefact data
  • Same manifest shape as the build-stream stub, but now populated with validation metrics and cutoff analysis payloads as well.
  • The steps array stores step ids, node types, node versions, statuses, and artifact hash links.
  • The artifacts array stores artifact ids, roles, artifact types, media types, and logical/physical hashes.
Sourcecardre/nodes/build/export.py and cardre/_evidence/models.py
